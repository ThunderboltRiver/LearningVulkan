# OSメモリシステムコール解説: `mmap` / `VirtualAlloc` vs `malloc`

このドキュメントでは、`DynamicArray<T>`で使用しているOSレベルのメモリ管理システムコール（`mmap`、`VirtualAlloc`）について、
標準ライブラリの`malloc`や`_aligned_malloc`との違い、使用理由、およびmacOS上での動作について説明する。

---

## 用語定義

本ドキュメントで使用する主要な用語を以下に定義する。

### フリーリスト（Free List）

`malloc`などのヒープアロケータが内部で管理するデータ構造の一つ。
`free()`で解放されたメモリブロックを連結リスト（リンクドリスト）として保持し、次の`malloc()`呼び出し時に再利用する仕組みである。

```
フリーリストの概念図:

  malloc()で確保    free()で解放      malloc()で確保
  ┌───────┐       ┌───────┐         ┌───────┐
  │使用中  │       │ 空き  │→next    │使用中  │
  └───────┘       └───────┘         └───────┘
                       ↓
                  ┌───────┐
                  │ 空き  │→next→NULL
                  └───────┘

  新たにmalloc()が呼ばれると、フリーリストから適切なサイズの空きブロックを探して返す。
  見つからない場合はOSから新しいメモリを確保する。
```

フリーリストの管理にはメタデータ（各ブロックのサイズ、次の空きブロックへのポインタ等）が必要であり、
これが`malloc`のオーバーヘッドの一因となる。

### 仮想アドレス空間と物理メモリ

現代のOSは**仮想メモリ**方式を採用しており、プログラムが扱うアドレスと実際のハードウェアメモリ（RAM）のアドレスは別物である。

- **仮想アドレス空間**: 各プロセスに与えられる「見かけ上の」メモリ空間。プログラムのコードが使用するポインタはすべて仮想アドレスを指す。プロセスごとに独立しており、あるプロセスの仮想アドレス`0x1000`と別のプロセスの`0x1000`は異なる物理メモリを指す。
- **物理メモリ（RAM）**: コンピュータに搭載された実際のメモリチップ。OSが管理し、複数のプロセスで共有される。
- **ページテーブル**: 仮想アドレスと物理アドレスの対応関係を保持するOSカーネル内の表。CPUのMMU（Memory Management Unit）がこの表を参照して、仮想アドレスを物理アドレスに変換する。

```
プロセスAの仮想アドレス空間          物理メモリ（RAM）
┌──────────────┐                ┌──────────────┐
│ 0x0000-0x0FFF│──────────────→│ 物理ページ 5  │
│ 0x1000-0x1FFF│──────────────→│ 物理ページ 12 │
│ 0x2000-0x2FFF│ (未マッピング) │              │
└──────────────┘                │              │
                                │              │
プロセスBの仮想アドレス空間        │              │
┌──────────────┐                │              │
│ 0x0000-0x0FFF│──────────────→│ 物理ページ 8  │
│ 0x1000-0x1FFF│──────────────→│ 物理ページ 20 │
└──────────────┘                └──────────────┘
        ↑                              ↑
  ページテーブルによるマッピング
```

`mmap`や`VirtualAlloc`は、この仮想アドレス空間にページ（通常4096バイト）単位で領域を追加し、
OSが物理メモリとの対応関係（マッピング）を設定する。

#### 仮想アドレス空間の最大容量はプロセス作成時に固定される

仮想アドレス空間のアドレス範囲（最大容量）は、**プロセス生成時にCPUアーキテクチャとOSによって固定的に決まる**。
プロセスの実行中にこの上限を変更することはできない。

| 環境 | ユーザー空間の仮想アドレス範囲 | 最大容量 |
|---|---|---|
| x86-64 Linux | `0x0000_0000_0000_0000` 〜 `0x0000_7FFF_FFFF_FFFF` | 約128TB |
| x86-64 Windows | `0x0000_0000_0000_0000` 〜 `0x0000_7FFF_FFFF_FFFF` | 約128TB |
| ARM64 macOS (Apple Silicon) | `0x0000_0000_0000_0000` 〜 `0x0000_FFFF_FFFF_FFFF` | 約256TB |

この上限は物理メモリ（RAM）の搭載量とは無関係であり、CPUが扱えるポインタのビット幅（x86-64では64ビットだが実際に使えるのは通常48〜52ビット）とOSの設計によって決まる。
`mmap`や`VirtualAlloc`で確保できる合計サイズはこの上限の範囲内に収める必要があるが、
現実的なアプリケーションではこの上限に達することはほとんどない。

### ゼロ初期化

メモリ領域のすべてのバイトが`0x00`に設定された状態を指す。
「実体が何も入っていない」という解釈は概念的には近いが、より正確には**すべてのビットが0に明示的に設定されている**状態である。

ここでの「メモリ領域のすべてのバイト」とは、**仮想アドレス空間上のバイト**のことであり、物理メモリ（RAM）のバイトそのものではない。
OSが**デマンドページング**（Demand Paging）という仕組みにより、`mmap`で仮想アドレス空間を確保した時点では実際の物理メモリは割り当てられず（遅延割り当て）、プログラムが実際にそのアドレスにアクセスした瞬間に初めて物理ページが割り当てられる。
その際、OSはセキュリティのために物理ページを必ずゼロクリアしてからプロセスに渡す。
そのためプログラムから見ると「確保した直後から全バイトが0」に見えるが、実際には物理メモリへの書き込みはアクセスのタイミングで初めて行われる。

```
mmap()直後の状態（仮想アドレス空間を確保しただけ）:
  仮想アドレス 0x7000〜0x8000: "マッピング済み（ゼロに見える）"
  物理ページ: まだ割り当てられていない（遅延割り当て）

プログラムが 0x7000 に初めてアクセスした時:
  1. CPUがページフォルト（アクセス違反ではなく正常な割り込み）を発生
  2. OSがゼロクリア済みの物理ページを割り当て
  3. ページテーブルに 0x7000 → 物理ページX の対応を記録
  4. プログラムの実行を再開 → 0x00 が読める
```

- **セキュリティ上の理由**: OSが新しいページをプロセスに割り当てる際、以前のプロセスが使用していたデータが残っていると情報漏洩になる。そのためOSはページを渡す前に必ずゼロクリアする。
- **C/C++での意味**: `int`なら`0`、`float`なら`0.0f`、ポインタなら`nullptr`に相当する値が格納されている。ただし`DynamicArray`では配置newで明示的にコンストラクタを呼ぶため、このゼロ初期化に依存した設計にはなっていない。

### インプレース拡張（In-place Expansion）

既にマッピングされているメモリ領域の**先頭アドレスを変えずに**、末尾方向にサイズを拡大する操作。

```
インプレース拡張のイメージ:

 拡張前:
  [データ領域 (8ページ)][ 未使用の仮想アドレス空間 ]
  0x7000              0xF000

 拡張後（インプレース成功時）:
  [     データ領域 (16ページ)     ][ 未使用 ]
  0x7000                         0x17000
  ↑ アドレス変わらず → 既存のポインタはすべて有効のまま

 インプレース不可の場合（アドレスが移動）:
  [別のマッピング]  →  [データ領域 (16ページ)]
  0x7000               0x20000
                        ↑ アドレスが変わる
```

`mremap`（Linux）はカーネルのページテーブル操作だけでこの拡張を行えるため、ユーザ空間でのデータコピーが不要で非常に高速である。
一方、`realloc`（`malloc`系）やmacOSのフォールバック処理では、新しい領域を確保してからデータをコピーする必要がある。

### アドレス空間の解放

`munmap`や`VirtualFree(MEM_RELEASE)`で行われるメモリ解放は、以下の処理を含む：

1. **仮想アドレスと物理メモリのマッピングを解除**する（ページテーブルのエントリを削除）
2. **物理メモリ（RAM）のページをOSの空きページプールに戻す**（他のプロセスが再利用可能になる）
3. **仮想アドレス範囲を未使用状態に戻す**（同じプロセスが後から別の用途に使える）

なお、「メモリの中身を消す（ゼロクリア）」はこの時点では必ずしも行われない。
実際にゼロクリアされるのは、そのページが次に別のプロセスに割り当てられるタイミングである（セキュリティのため）。

### POSIX（Portable Operating System Interface）

POSIX（ポシックス）はIEEE（米国電気電子学会）が策定した、**Unix系オペレーティングシステムの共通インタフェース標準規格**（IEEE Std 1003）である。
カーネルの規格そのものではなく、**OSがアプリケーションに提供すべきAPI（システムコール、コマンド、シェルの仕様等）を定めた標準仕様**である。

- **目的**: 異なるUnix系OS間でソースコードレベルの移植性を保証すること。POSIXに準拠したプログラムは、Linux・macOS・FreeBSD等のPOSIX準拠OSで再コンパイルするだけで動作する（理論上）。
- **POSIX準拠OS**: Linux、macOS、FreeBSD、Solaris、AIX等のUnix系OSはPOSIXに準拠（または大部分を実装）している。WindowsはPOSIX非準拠だが、WSL（Windows Subsystem for Linux）を通じてPOSIX APIを提供する。
- **`mmap`とPOSIX**: `mmap`、`munmap`、`mprotect`、`sysconf`等はPOSIX標準で規定されたシステムコールであり、POSIX準拠のすべてのOSで利用できる。`mremap`はPOSIX標準には含まれず、Linux独自の拡張である。

```
POSIX標準の位置づけ:

  アプリケーション
       ↓  （POSIX API: mmap, open, read, write, fork, exec, ...）
  ─────────────────── POSIX標準が定めるインタフェース境界
       ↓
  OSカーネル（実装はOS依存）
  - Linux: Linux Kernel
  - macOS: XNU (Darwin)
  - FreeBSD: FreeBSD Kernel
```

### ユーザー空間とカーネル空間（User Space / Kernel Space）

現代のOS（Linux・macOS・Windowsを含むすべての主要OS）は、仮想アドレス空間を**ユーザー空間**と**カーネル空間**の2つの領域に分けて管理している。これはLinux固有の概念ではなく、OSのセキュリティ・安定性を保つために広く採用されている設計である。

- **ユーザー空間（User Space）**: アプリケーションプログラム（ブラウザ、ゲーム、C++プログラム等）が動作するアドレス領域。`malloc`、`mmap`、`VirtualAlloc`等で確保されるメモリはすべてユーザー空間に属する。ユーザー空間のプログラムは直接ハードウェアや他プロセスのメモリにアクセスできず、カーネルを介してのみ操作できる。

- **カーネル空間（Kernel Space）**: OSカーネルが動作するアドレス領域。デバイスドライバ、ページテーブル管理、システムコール処理等はここで行われる。カーネル空間はユーザー空間から直接アクセスできない（試みると「セグメンテーションフォルト」等のエラーになる）。

```
仮想アドレス空間の分割（x86-64 Linuxの例）:

  高位アドレス
  ┌─────────────────────────────┐  0xFFFF_FFFF_FFFF_FFFF
  │                             │
  │      カーネル空間            │  ← OSカーネル、ページテーブル、
  │  (Kernel Space)             │     デバイスドライバ等が動作
  │                             │
  ├─────────────────────────────┤  0xFFFF_8000_0000_0000
  │     （未使用のアドレス範囲）   │
  ├─────────────────────────────┤  0x0000_8000_0000_0000
  │                             │
  │      ユーザー空間            │  ← アプリケーションのコード・データ・
  │  (User Space)               │     スタック・ヒープ・mmap領域等
  │                             │
  └─────────────────────────────┘  0x0000_0000_0000_0000
  低位アドレス
```

`mremap`（Linux）の説明にある「ユーザ空間でのデータコピーが不要」とは、アプリケーション（ユーザー空間）がデータをコピーするCPU命令を実行しなくて済む、という意味である。`mremap`ではカーネル空間でページテーブルを書き換えるだけでメモリの「移動」が完了するため、大量データのコピーによるCPU時間とメモリ帯域の消費を避けられる。

| OS | ユーザー空間 | カーネル空間 |
|---|---|---|
| Linux | ✅ あり | ✅ Linux Kernel |
| macOS | ✅ あり | ✅ XNU (Darwin Kernel + Mach Microkernel) |
| Windows | ✅ あり | ✅ Windows NT Kernel |

## 1. `malloc` / `_aligned_malloc` とは

### `malloc`
- C標準ライブラリが提供する汎用メモリ割り当て関数。
- 内部でOSのシステムコール（`brk`/`sbrk`や`mmap`など）を呼び出してメモリを確保するが、その詳細はユーザから隠蔽されている。
- 小さなブロックの割り当て・解放を効率よく行うためにフリーリスト（→「用語定義: フリーリスト」参照）等のデータ構造を内部に持つ。
- `free()`で解放しても、OSにメモリが即座に返されるとは限らない（ランタイムが再利用のためにキャッシュすることがある）。

### `_aligned_malloc`（Windows固有）
- 指定したアライメント境界にメモリを割り当てる`malloc`の亜種。
- POSIX環境では`posix_memalign`や`aligned_alloc`(C11)が同等の機能を提供する。
- `malloc`と同様、内部では汎用ヒープアロケータを通じてメモリを管理する。

**`malloc`系の特徴まとめ:**
| 項目 | `malloc` / `_aligned_malloc` |
|---|---|
| 粒度 | バイト単位（任意サイズ） |
| メモリソース | ランタイムのヒープ（内部でOSコールを使用） |
| オーバーヘッド | ヒープ管理メタデータ（ヘッダ、フリーリスト等） |
| 解放時の挙動 | ランタイムがキャッシュ。OSに即座には返さないことが多い |
| アドレス空間の制御 | なし（ランタイム任せ） |

## 2. `mmap` とは（POSIX）

`mmap`はPOSIXオペレーティングシステム（→「用語定義: POSIX」参照）が提供するシステムコールで、**プロセスの仮想アドレス空間（→「用語定義: 仮想アドレス空間と物理メモリ」参照）にメモリ領域を直接マッピング**する。ここでの「メモリ領域」は物理メモリ（RAM）そのものではなく、仮想アドレス空間上の領域であり、OSが必要に応じて物理メモリとの対応関係を設定する。

```c
#include <sys/mman.h>

void* mmap(
    void*  addr,    // 割り当て先アドレスのヒント（通常はnullptr）
    size_t length,  // 割り当てるバイト数
    int    prot,    // メモリ保護属性（PROT_READ | PROT_WRITE 等）
    int    flags,   // 割り当てフラグ（MAP_PRIVATE | MAP_ANONYMOUS 等）
    int    fd,      // ファイルディスクリプタ（匿名マッピングの場合は -1）
    off_t  offset   // ファイルオフセット（匿名マッピングの場合は 0）
);
```

### 主な用途
1. **ファイルマッピング**: ファイルの内容をメモリに直接マッピングし、`read`/`write`システムコールなしにアクセスする。
2. **匿名マッピング**（`MAP_ANONYMOUS`）: ファイルに紐づかないゼロ初期化（→「用語定義: ゼロ初期化」参照）されたメモリ領域を確保する。`DynamicArray`ではこちらを使用している。

### 解放: `munmap`

```c
int munmap(void* addr, size_t length);
```

`munmap`は指定されたメモリ領域をプロセスのアドレス空間から解除し、**直ちにOSに物理メモリを返す**。

### Linux固有: `mremap`

```c
#include <sys/mman.h>

void* mremap(
    void*  old_address,  // 既存のマッピングの先頭アドレス
    size_t old_size,     // 既存のマッピングのサイズ
    size_t new_size,     // 新しいサイズ
    int    flags         // MREMAP_MAYMOVE 等
);
```

`mremap`は**Linux固有**のシステムコールで、既存のメモリマッピングのサイズを変更する。
- `MREMAP_MAYMOVE`フラグにより、仮想アドレス空間に余裕がある場合はアドレスを変えずにインプレース拡張（→「用語定義: インプレース拡張」参照）できる。
- インプレース拡張が可能な場合、データのコピーは一切発生しないため非常に高速。
- インプレース拡張が不可能な場合は、新しいアドレスにマッピングを移動する（カーネルがページテーブルを書き換えるだけなので、ユーザ空間でのデータコピーは不要）。

## 3. `VirtualAlloc` とは（Windows）

`VirtualAlloc`はWindows APIが提供するメモリ管理関数で、**仮想アドレス空間を直接操作**する。

```c
#include <windows.h>

LPVOID VirtualAlloc(
    LPVOID lpAddress,         // 割り当て先アドレス（通常はNULL）
    SIZE_T dwSize,            // 割り当てるバイト数
    DWORD  flAllocationType,  // 割り当て種別（MEM_RESERVE, MEM_COMMIT 等）
    DWORD  flProtect          // メモリ保護属性（PAGE_READWRITE 等）
);
```

### 2段階のメモリ管理
Windowsの仮想メモリシステムは**予約（Reserve）**と**コミット（Commit）**の2段階モデルを採用している：

1. **`MEM_RESERVE`**: 仮想アドレス空間を予約するが、物理メモリやページファイルの割り当ては行わない。
2. **`MEM_COMMIT`**: 予約された領域に物理メモリ（またはページファイル）を割り当てる。

`DynamicArray`では`MEM_COMMIT | MEM_RESERVE`を同時に指定して、予約とコミットを一度に行っている。

### 解放: `VirtualFree`

```c
BOOL VirtualFree(
    LPVOID lpAddress,    // 解放するメモリの先頭アドレス
    SIZE_T dwSize,       // 解放するバイト数（MEM_RELEASEの場合は0）
    DWORD  dwFreeType    // MEM_RELEASE または MEM_DECOMMIT
);
```

`MEM_RELEASE`で呼び出すと、予約されたアドレス空間全体を解放し、物理メモリをOSに返す（→「用語定義: アドレス空間の解放」参照）。具体的には、仮想アドレスと物理メモリのマッピングが解除され、物理メモリのページがOSの空きプールに戻されて他のプロセスが使用可能な状態になる。メモリの中身のゼロクリアは解放時ではなく、次にそのページが別のプロセスに割り当てられるタイミングで行われる。

## 4. `malloc` との主要な違い

| 比較項目 | `malloc` / `_aligned_malloc` | `mmap` / `VirtualAlloc` |
|---|---|---|
| **抽象レベル** | Cランタイムライブラリ（ユーザ空間） | OSカーネルのシステムコール |
| **メモリ粒度** | バイト単位（任意サイズ） | ページ単位（通常4096バイト） |
| **割り当て単位** | 小さいブロックに最適化 | 大きなブロック向け |
| **メタデータ** | ヒープ管理のオーバーヘッドあり | オーバーヘッドなし（カーネルが管理） |
| **解放時** | ランタイムがキャッシュ、OSに返さないことがある | `munmap`/`VirtualFree`で**即座にOSに返る** |
| **ゼロ初期化** | 保証なし（`calloc`を除く） | `mmap`(匿名)と`VirtualAlloc`はゼロ初期化保証 |
| **アドレス空間制御** | 不可 | 予約・コミットの分離が可能（`VirtualAlloc`） |
| **インプレース拡張** | `realloc`はコピーが必要な場合がある | `mremap`(Linux)はコピーなしで拡張できる可能性がある |
| **アライメント** | `malloc`は通常`max_align_t`に揃う | ページ境界（4096バイト等）に常に揃う |

### 図解: メモリ割り当ての流れ

```
malloc の場合:
  ユーザコード → malloc() → Cランタイムヒープ → [内部で mmap/brk] → カーネル
                              ↑
                        ヒープ管理メタデータ
                        フリーリスト等

mmap / VirtualAlloc の場合:
  ユーザコード → mmap() / VirtualAlloc() → カーネル（直接）
                                            ↑
                                   中間レイヤーなし
```

## 5. `DynamicArray`で`mmap`/`VirtualAlloc`を使用する理由

### 5.1. バンプアロケータのバックボーンとしての適性

`DynamicArray`は将来的にバンプアロケータ（リニアアロケータ）の基盤として使用することを想定している。
バンプアロケータは大きなメモリブロックをOSから直接確保し、その中で個々のオブジェクトを先頭から順に配置していく。
この用途では：

- **大きな連続メモリブロック**が必要 → `mmap`/`VirtualAlloc`はページ単位で大きなブロックを効率的に確保できる
- **ヒープのフラグメンテーションを回避**したい → `malloc`のフリーリストを経由しないため、フラグメンテーションが発生しない
- **解放時に確実にOSに返したい** → `munmap`/`VirtualFree`は即座にメモリをOSに返す

### 5.2. `mremap`によるインプレース拡張の利点

Linux上では`mremap`により、配列の拡張時にデータのコピーを回避できる可能性がある。
通常の`realloc`（`malloc`系）では拡張時にほぼ必ずデータのコピーが発生するが、
`mremap`はカーネルのページテーブル操作のみで拡張を実現できるため、
大量のデータを保持する配列の拡張が非常に高速になる。

### 5.3. `PlacementStackAllocator`の制約を超えるため

現在のコードベースでは`Span<T>`が`PlacementStackAllocator`を使用しているが、
これは固定サイズのスタック領域（`malloc`で確保された12MBブロック）に制限されている。
`PlacementStackAllocator.h`と`Span.h`の両方にTODOコメントでこの制約が記されており、
`DynamicArray`はその制約を解消するための第一歩となる。

### 5.4. メモリのオーバーヘッド削減

`malloc`は各割り当てにヒープ管理のためのメタデータ（通常16〜32バイト）を付加する。
`mmap`/`VirtualAlloc`はカーネルがページテーブルでメモリを管理するため、
ユーザ空間でのメタデータオーバーヘッドがない。
大きなメモリブロックを少数回だけ確保する用途ではこの差が有利に働く。

## 6. macOSでの`mmap`の動作

### `mmap`はmacOS上で動作する

`mmap`はPOSIX標準（→「用語定義: POSIX」参照）のシステムコールであり、**macOS（Darwin）でも完全に動作する**。
macOSはBSD系のカーネル（XNU）を使用しており、POSIX互換のシステムコールを広くサポートしている。

`DynamicArray`で使用している以下のフラグ・関数はすべてmacOS上で利用可能：

| 機能 | macOS対応 | 備考 |
|---|---|---|
| `mmap` with `MAP_PRIVATE \| MAP_ANONYMOUS` | ✅ 対応 | `MAP_ANONYMOUS`はmacOSでは`MAP_ANON`としても定義されている |
| `munmap` | ✅ 対応 | POSIX標準 |
| `mprotect` | ✅ 対応 | POSIX標準 |
| `mremap` | ❌ **非対応** | Linux固有のシステムコール |
| `sysconf(_SC_PAGESIZE)` | ✅ 対応 | POSIX標準 |

### macOSにおける`mremap`の非対応

`mremap`はLinuxカーネル固有の拡張であり、macOS（およびFreeBSD等のBSD系OS）には存在しない。
そのため、`DynamicArray`のコードではプリプロセッサ分岐で対応している：

```cpp
#if defined(__linux__)
    // Linux: mremap でインプレース拡張を試みる
    void* ptr = mremap(oldPtr, oldBytes, newBytes, MREMAP_MAYMOVE);
#elif defined(_WIN32)
    // Windows: VirtualAlloc + memcpy + VirtualFree
    ...
#else
    // macOS等のPOSIX環境: mmap（新規確保）+ memcpy + munmap（旧領域解放）
    void* newPtr = mmap(nullptr, newBytes, PROT_READ | PROT_WRITE,
                        MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
    std::memcpy(newPtr, oldPtr, oldBytes);
    munmap(oldPtr, oldBytes);
#endif
```

macOS上では`mremap`が使えないため、新しいメモリ領域を`mmap`で確保し、
既存データを`memcpy`でコピーしてから旧領域を`munmap`で解放するフォールバック処理を行う。
これは`realloc`の内部動作と概念的に同等だが、ヒープ管理のオーバーヘッドがない点が異なる。

### macOSでのMach VM APIを使った実現可能性

macOSは**Mach VM API**（`<mach/vm_map.h>`）というカーネルレベルの低位メモリ管理インタフェースを提供しており、
`mmap`の内部では実際にこのMach VM APIが呼び出されている。
Mach VM APIを直接使用することで、`mmap`では実現できない操作が可能になる。

#### 主要なMach VM API関数

| 関数 | 対応する`mmap`系の操作 | 説明 |
|---|---|---|
| `vm_allocate()` | `mmap(MAP_ANON)` | 匿名メモリ領域を確保する |
| `vm_deallocate()` | `munmap()` | メモリ領域を解放する |
| `vm_remap()` | `mremap()`（Linuxのみ） | 既存のマッピングを別のアドレスにコピー/移動する |
| `vm_copy()` | `memcpy()`（COW最適化あり） | コピーオンライト（COW）で仮想的にメモリをコピーする |
| `mach_vm_allocate()` | `mmap(MAP_ANON)` | `vm_allocate`の64ビット版 |

#### `vm_remap`による`mremap`相当の操作

`vm_remap`を使うと、既存のメモリマッピングを**コピーオンライト（COW）**で別のアドレスに複製できる。
ただしLinuxの`mremap`と異なり、「同じ仮想アドレスのままサイズを拡張する」インプレース拡張ではなく、
新しいアドレスに複製してから旧領域を解放するという手順になる。

```c
#include <mach/mach.h>

// macOS での vm_remap を使ったリサイズの概念
vm_address_t new_addr = 0;       // OSに任せる
vm_prot_t cur_prot, max_prot;

kern_return_t kr = vm_remap(
    mach_task_self(),            // 自プロセス
    &new_addr,                   // 新しいアドレス（出力）
    new_size,                    // 新しいサイズ
    0,                           // アラインメント
    TRUE,                        // 任意のアドレスに配置を許可
    mach_task_self(),            // コピー元プロセス
    (vm_address_t)old_ptr,       // コピー元アドレス
    FALSE,                       // FALSE = copy（COW）
    &cur_prot, &max_prot,
    VM_INHERIT_DEFAULT
);
// 追加領域は別途 vm_allocate で確保して old_addr に続けて配置するか、
// new_addr 以降に新規確保した領域を結合する必要がある
```

#### `DynamicArray`でMach VM APIを使う場合の考慮点

| 比較項目 | 現在の実装（`mmap`フォールバック） | Mach VM APIを使う場合 |
|---|---|---|
| **コード複雑度** | 低（POSIXコードと統一） | 高（macOS専用コードが必要） |
| **移植性** | POSIX準拠OSで共通 | macOS専用 |
| **拡張時のコピー** | `memcpy`でデータコピーが発生 | `vm_remap`（COW）でページコピーは遅延 |
| **実用上のメリット** | 限定的 | 大容量データの拡張でCOWの恩恵が得られる可能性あり |

現在の`DynamicArray`の用途規模では`mmap`フォールバックで十分であるが、
macOS上で非常に大きな配列を頻繁に拡張する場合は、Mach VM APIを使ったmacOS専用最適化が選択肢となりうる。

## 7. まとめ

| 質問 | 回答 |
|---|---|
| `mmap`/`VirtualAlloc`とは？ | OSカーネルが提供するメモリマッピングのシステムコール。仮想アドレス空間を直接操作する |
| `malloc`との違いは？ | `malloc`はCランタイムのヒープマネージャを経由する。`mmap`/`VirtualAlloc`はカーネルに直接要求する |
| なぜ`malloc`ではなくこれらを使用するのか？ | バンプアロケータの基盤として大きなブロックを効率的に管理し、ヒープフラグメンテーションを回避し、`mremap`(Linux)でコピーなし拡張を実現するため |
| `mmap`はmacOSで動作するか？ | ✅ 動作する（POSIX標準）。ただし`mremap`はLinux固有のため非対応。macOSではフォールバック処理で対応 |
